# 語言練習紀錄上傳：實作前確認問題

本文件記錄 Smart English Trainer 對接 `LANGUAGE_PRACTICE_UPLOAD_SPEC.md` 前，需要確認的決策與外部資訊。

請直接在每一題的「確認結果」後填寫答案。尚未確認的項目可保留 `待確認`。

## 一、請專案負責人確認

### 1. 第一版的上傳方式

第一版要採用哪一種方式？

- A. 只有使用者按下「立即同步」才上傳
- B. 每天自動排程上傳 1～2 次
- C. 手動同步與自動排程都要支援

建議：第一版先做 A，穩定後再增加 C。

確認結果：第一版採用 **A，以使用者按下「立即同步」後批次上傳**。程式結構需保留排程呼叫同一個同步服務的能力，待手動同步驗證穩定後，再擴充為 C。預期正常使用頻率為每天完成練習後上傳一至兩次，不要求逐題即時上傳。

### 2. 要納入哪些練習類型

目前專案包含：

- 句子拼寫練習
- 句子重組練習
- TOEIC Part 5
- TOEIC Part 2

第一版是否四種練習都要建立及上傳紀錄？還是只處理句子拼寫與重組？

確認結果：第一版四種都納入，包括句子拼寫、句子重組、TOEIC Part 5 與 TOEIC Part 2。四種練習都轉換成相同的事件格式上傳；某些題型沒有的欄位可以傳 `null` 或省略，但 `event_id`、`sentence_id`、`exercise_type`、`is_correct` 與 `occurred_at` 必須提供。

### 3. `exercise_type` 的正式值

建議固定使用下列值：

```text
spelling
scramble
toeic_part5
toeic_part2
```

平台是否接受這些值？

確認結果：平台接受以下固定值：`spelling`、`scramble`、`toeic_part5`、`toeic_part2`。第一版採固定 enum，不接受任意新值；未來增加題型時再同步擴充 enum。這能避免相同題型因拼字差異被拆成不同統計類別。

### 4. `sentence_id` 的上傳格式

本機題目 ID 目前是整數，而且不同類型題目分散於不同資料表。建議上傳時轉成有前綴的字串，例如：

```text
en-sentence-123
vi-sentence-123
toeic-part5-456
toeic-part2-789
```

是否同意使用這種格式？若平台另有格式要求，請提供格式。

確認結果：同意使用帶前綴的字串，格式採 `{language}-{source_type}-{local_id}`。句子題例如 `en-sentence-123`、`vi-sentence-123`；TOEIC 題採 `en-toeic-part5-456`、`en-toeic-part2-789`。第一版最大長度定為 120 字元，只允許英文字母、數字、連字號 `-`、底線 `_`、句點 `.` 與冒號 `:`。平台將 `sentence_id` 視為來源系統識別碼，不以它判斷一次練習是否重複。

### 5. 是否保存及上傳 `user_answer`

是否允許保存並上傳使用者的實際答案？

需要一併確認：

- 使用者答案是否可能包含姓名、電話或其他敏感內容？
- TOEIC 選擇題要傳選項代碼（例如 `A`），還是完整選項文字？
- 若不需要分析答案內容，是否完全不傳 `user_answer`？

確認結果：第一版允許保存及上傳 `user_answer`，用途是日後檢查錯題及分析學習過程；此欄位為選填。句子題傳實際輸入文字，TOEIC 選擇題只傳選項代碼 `A`、`B`、`C`、`D`，不重複傳完整選項文字。語言系統介面應提醒使用者不要輸入個資；平台也不應把答案內容寫入一般應用程式 log。若特定練習沒有答案內容，可省略或傳 `null`。

### 6. `device_id` 的生命週期

建議第一次啟動時產生 UUID，永久保存在本機；除非重設應用程式，否則不改變。

請確認 `device_id` 要代表：

- A. 一台電腦
- B. 一次應用程式安裝
- C. 一位使用者

建議：B，一次安裝一個 ID；使用者身分仍由平台 Token 決定。

確認結果：採用 **B，一次應用程式安裝一個 UUID**。第一次啟動時建立並永久保存；重新啟動或升級程式不得改變，只有清除資料或重新安裝才重新產生。使用者身分一律由平台 Token 決定，`device_id` 不代表使用者，也不能作為授權依據。

### 7. 時區處理

是否將預設時區設為 `Asia/Bangkok`？

每筆資料仍會保存含 UTC offset 的時間，例如：

```text
2026-07-12T23:30:00+07:00
```

若程式在其他時區執行，應使用電腦當下時區，還是仍固定使用 `Asia/Bangkok`？

確認結果：預設時區為 `Asia/Bangkok`。每筆 `occurred_at` 必須包含當下正確 UTC offset，並同時傳送有效的 IANA `timezone`。若電腦確實在其他時區執行，使用電腦當下的 IANA 時區，不強制改成 Bangkok；如果無法可靠取得 IANA 時區，第一版可退回 `Asia/Bangkok`，但介面需允許使用者確認設定。

### 8. 歷史資料是否需要補建及上傳

目前 `trainer.db` 已經存在舊的練習紀錄，但這些資料沒有：

- `event_id`
- 完整 `occurred_at` 時區資訊
- `exercise_type`
- `user_answer`
- 題目文字快照

第一版要：

- A. 只同步修改完成後的新紀錄
- B. 嘗試轉換並補傳舊紀錄

建議：A，避免用不完整資料推測歷史事件。

確認結果：採用 **A，只同步新版紀錄機制上線後建立的新紀錄**。舊資料不自動推測或補傳，以免時間、題型與題目快照不完整造成錯誤統計。若未來需要歷史匯入，另做一次性匯入規格，並標記資料來源與推測欄位。

### 9. 同步失敗的使用者介面

是否需要在畫面顯示：

- 待同步筆數
- 上次同步時間
- 同步成功／失敗筆數
- 逐筆錯誤原因
- 「重新同步」按鈕

建議第一版至少顯示待同步筆數、上次同步時間、結果摘要及重新同步按鈕。

確認結果：第一版至少顯示待同步筆數、上次成功同步時間、本次成功／重複／失敗筆數，以及「立即同步／重新同步」按鈕。逐筆錯誤原因需保存於本機，介面可先以可展開的錯誤明細或錯誤紀錄頁呈現，不必阻塞第一版主流程。網路錯誤與平台暫時錯誤的資料維持 `pending`；欄位驗證失敗才標為 `failed`。

### 10. 單日及單次資料量

請估計：

- 單日最多可能產生多少筆練習紀錄？
- 是否接受依規格每批最多上傳 100 筆，超過時自動分批？

確認結果：平台先以單日最多 2,000 筆作為第一版容量目標。每批最多 100 筆，超過時由語言系統自動分批，依建立時間由舊到新上傳。此數量不是每日硬性拒絕上限；若實際使用可能超過 2,000 筆，對接方需提前告知以進行壓力測試。

### 11. 近期語言範圍

第一版是否只需要：

```text
en
vi
```

近期是否預計加入泰文、日文或其他語言？

確認結果：第一版只支援 `en` 與 `vi`。近期其他語言不納入本期，但資料庫欄位與程式碼不可把語言限制寫死成只能存兩個字元；未來新增語言時，以 ISO 639-1 小寫代碼擴充允許清單。

## 二、需要 Personal Learning & Workspace 平台端提供或確認

### 12. 正式 API 位址

請提供開發／測試及正式環境的 Base URL，例如：

```text
測試：https://staging.example.com
正式：https://example.com
```

並確認端點是否確定為：

```http
POST /api/v1/study/practice-logs
```

確認結果：開發／本機測試 Base URL 暫定為 `http://127.0.0.1:5050`，端點確定規劃為 `POST /api/v1/study/practice-logs`。目前此端點尚未實作，因此現在還不能進行實際串接。正式環境 Base URL 尚未決定，部署完成後另行提供；客戶端必須將 Base URL 設為可設定值，不可寫死。

### 13. Bearer Token 的取得方式

請確認：

- Token 由平台管理者手動產生，還是經由登入 API 取得？
- Token 是否會過期？
- 若會過期，如何更新？
- 是否需要 `study:write` scope？
- 測試用 Token 如何取得？

確認結果：Token 由已登入學習平台的使用者在設定頁手動建立，不透過帳號密碼登入 API 取得。第一版 Token 預設不自動過期，但可由使用者隨時撤銷；平台可選擇設定到期日。權限固定需要 `study:write` scope。完整 Token 只在建立當下顯示一次，平台資料庫只保存雜湊。測試用 Token 將在 API Token 功能及測試環境完成後，由平台設定頁建立；目前尚無可用測試 Token。

### 14. Token 的本機保存方式

本專案應將 Token 保存在哪裡？

- A. Windows Credential Manager 等系統安全儲存區
- B. 僅從環境變數讀取
- C. 本機設定檔
- D. 由使用者每次輸入

基於安全性，不建議將明文 Token 放入 `config.json`、SQLite、Git 或應用程式 log。

確認結果：Windows 正式版本優先採用 **A，Windows Credential Manager**。為方便開發及自動化測試，也允許 **B，環境變數** 作為覆寫來源。不得將明文 Token 寫入 `config.json`、SQLite、Git、錯誤回報或應用程式 log；不採用每次輸入。若 Credential Manager 暫時無法在第一版完成，開發版可先只支援環境變數，但不得以明文設定檔替代。

### 15. API 欄位與驗證規則

請平台確認：

- `event_id` 是否接受 UUID v4？
- `sentence_id` 最大長度及允許字元？
- `exercise_type` 是否為自由字串或固定 enum？
- `sentence_text`、`translation_text`、`user_answer` 的最大長度？
- `timezone` 是否必須是有效 IANA 時區？
- `events` 是否允許空陣列？
- 每批是否確定最多 100 筆？
- Request body 最大容量？

確認結果：平台規則如下：

- `event_id` 接受標準 UUID v4 字串，長度 36；第一版只接受 UUID v4。
- `sentence_id` 最大 120 字元，允許英文字母、數字、`-`、`_`、`.`、`:`。
- `exercise_type` 是固定 enum：`spelling`、`scramble`、`toeic_part5`、`toeic_part2`。
- `sentence_text` 最大 10,000 字元且必填；`translation_text` 最大 10,000 字元；`user_answer` 最大 10,000 字元。
- `timezone` 建議必填，且必須是有效 IANA 時區；`occurred_at` 必填並必須包含 UTC offset。
- `events` 不允許空陣列。
- 每批最多 100 筆。
- Request body 最大 1 MiB；超過時回傳 `413`。
- `audio_play_count` 必須是 0～10,000 的整數；`is_correct`、`revealed_answer` 必須是真正的 JSON boolean。
- 未知的額外欄位第一版忽略，以保留向後相容性，但不保存。

### 16. 部分成功的 HTTP 狀態

規格建議批次已處理時一律回傳 HTTP `200`，並由回應內的 `results` 表示個別項目的 `created`、`duplicate` 或 `rejected`。

請確認部分成功時是否確定使用 HTTP `200`，而不是 `207`、`400` 或其他狀態。

確認結果：確定使用 HTTP `200` 表示「批次請求已成功解析並完成逐筆處理」，包括全部成功、全部重複或部分成功。客戶端必須讀取每筆 `results[].status`。整個 request 結構無法解析時才回 `400`；驗證失敗的單筆事件回 `rejected`，不把整批改成 `400`。不使用 `207`。

### 17. 重複事件的判定

請確認 `event_id` 的唯一範圍：

- A. 全平台全域唯一
- B. 同一平台使用者內唯一
- C. 同一使用者與 `device_id` 組合內唯一

建議：至少在同一使用者內唯一；如果使用 UUID，也可以做全平台唯一。

另外請確認：相同 `event_id` 但內容不同時，平台要回傳 `duplicate`，還是回傳衝突錯誤？

確認結果：採用 **B，同一平台使用者內 `event_id` 唯一**，資料庫唯一約束為 `(user_id, event_id)`。不同裝置也不得替同一使用者重用 event UUID；`device_id` 用於追蹤來源，不放入去重唯一鍵。相同 `event_id` 且內容相同時回 `duplicate` 及原本的 `log_id`；相同 `event_id` 但任何核心內容不同時回 `conflict`，不得覆寫既有紀錄，並在該項目提供錯誤原因。此項仍以 HTTP `200` 回傳逐筆結果。

### 18. 每日彙整的時區規則

請確認每日歸屬日期的計算方式：

- 直接使用 `occurred_at` 內的 UTC offset
- 將 `occurred_at` 轉換成 `timezone` 指定時區
- 統一轉換成平台使用者設定的時區

也請確認夏令時間或使用者旅行到其他時區時的處理規則。

確認結果：平台先驗證 `timezone`，再將 `occurred_at` 代表的時間點轉換到該 IANA 時區，以轉換後的當地日期作為 `study_date`。不能只截取時間字串中的日期。夏令時間交由 IANA timezone 規則處理；使用者旅行時，該筆事件傳當時所在地時區，即歸入當地學習日。若 timezone 與 UTC offset 在該時間點明顯不一致，該事件回 `rejected`，避免歸錯日期。

### 19. 使用者隔離與帳號切換

規格指出使用者身分由 Token 決定。請平台確認：

- 不同 Token 是否可能代表同一使用者？
- 同一台電腦更換 Token 後，尚未同步的資料應歸屬哪位使用者？
- API 是否會在回應中提供使用者或 Token 識別資訊，供客戶端防止傳錯帳號？

確認結果：不同 Token 可以屬於同一平台使用者，例如家用電腦與筆電各一枚 Token。尚未同步的事件必須綁定「建立事件當下所選的平台帳號識別」，不能在更換 Token 後自動上傳到新帳號；若本機目前尚未設計多帳號資料邊界，存在 pending 資料時應禁止直接換 Token，或要求先同步／明確捨棄。平台另提供 `GET /api/v1/auth/me`，回傳非敏感的 `user_id`、`username`、`display_name`、Token prefix 與 scopes，客戶端在保存 Token 時先確認帳號，並在每次同步前防止帳號不符。

### 20. 測試環境與驗收資料

請平台提供：

- 可重設資料的測試環境
- 有效 Token
- 無效或已撤銷 Token
- 沒有 `study:write` 權限的 Token
- 查看已接收練習明細及每日彙整結果的方法

沒有上述測試條件，無法完整驗證規格第 12 節的驗收項目。

確認結果：上述測試條件都列為平台端實作及驗收工作的必要交付物，但目前尚未完成，因此現在無法提供測試 URL 或 Token。平台端完成第一版後需提供：可清理的本機／測試資料庫、有效 `study:write` Token、已撤銷 Token、缺少 scope Token，以及登入平台後查看練習明細與每日彙整的頁面。正式對接驗收不能直接使用正式資料庫。

